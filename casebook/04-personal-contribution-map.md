# 个人贡献与能力映射

## 核心角色

- AI Native 产品经理 / 项目统筹：把 OPC 的租户、渠道、商品、供应商、分润、履约、证据链和诊断工作流拆成可确认、可交付、可追踪的材料。
- 复杂需求拆解：把会议、XMind、Figma Make 包、截图 OCR、Excel 和前端包整理成业务模块、状态口径和开发指令。
- AI 工作流设计：把一次性人工整理流程转成 Skill、证据库、每日增量、状态看板和诊断端。
- 项目证明口径：公开仓库展示的是“我做过什么、怎么做、产出了什么、能证明什么能力”，不是原始业务资料归档。

## 能力映射

| 能力 | OPC 证据 | 可用于简历的表达 |
|---|---|---|
| 复杂业务架构 | OPC 系统需求总包、业务模块总表、功能需求清单 | 梳理多租户、渠道、商品、分润和履约链路，形成可交付 PRD 和开发指令 |
| Figma Make 协作治理 | 六包/五包工作区规则、包级归属、页面差异清单 | 建立工作区规则和验收口径，降低“草稿/页面/闭环”误判 |
| 证据化项目管理 | 证据库索引、每日增量、状态表、诊断端 | 把分散材料整理成可追踪、可复盘、可诊断的项目资产 |
| AI 工作流产品化 | Skill 工作流、董事长诊断端、每日技术同步 H5 | 将人工判断流程拆成可复用的 Agent 工作流和展示页面 |
| 公开表达与脱敏 | 本仓库的 casebook、demo、公开素材库 | 能把复杂内部项目整理成对外可读、可证明、不过度暴露的作品集 |

## 不公开的边界

- 不公开原始聊天记录、会议音频、完整日志、API 返回、Feishu 原始同步结果。
- 不公开本机绝对路径、私有仓库地址、字段 ID、凭证、手机、邮箱。
- 不把大 zip、模型、音频和临时目录放进 GitHub。

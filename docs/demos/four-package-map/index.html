OPC 四包功能归属展示说明 · 按现有包内材料整理

平台配置、平台配置_Copy、营销推广、日常运营：功能归属与端侧展示

这份 H5 按现有前端包盘点、包级对比清单和工作区规则整理。重点说明每个功能在什么场景下存在、由哪个包主责、PC 端和小程序端分别怎么看。

范围：4 个包 依据：2026-05-22 最新前端包盘点 口径：有材料/有组件不等于业务闭环完成
先看结论 端侧口径 选品与采购 场景导航 平台配置 平台配置_Copy 营销推广 日常运营 功能归属表 展示判断规则

一、先看结论

四个包可以先按一句话理解：

平台配置 平台规则中心

负责平台级规则、模板、系统设置、分润/结算规则、供应商准入入口。

平台配置_Copy 商品从采购到上架的工作台

负责进货采购、上架配置、库存、CPS 履约、订单、批量操作。

营销推广 对外推广和传播中心

负责 AI 素材、推广任务、活动、裂变、视频号推客、推广效果。

日常运营 每天怎么运营的工作台

负责今日待办、签到、任务、运营标签维护、数据看板、活动执行跟进。

展示判断法：如果是在“定规则”，优先看平台配置；如果是在“把货买来并上架”，看平台配置_Copy；如果是在“拉人传播和推广”，看营销推广；如果是在“每天运营人员要做什么”，看日常运营。

二、PC 端 / 小程序端展示口径

这四个包里同时出现 PC 端管理页面和小程序端页面。文档里统一按下面口径展示，避免只按小程序端理解。

端侧 主要用途 常见页面/组件证据 展示时怎么理解
PC 端 偏后台、配置、管理、审核、数据看板 MainApp、Dashboard、EnhancedDashboard、ProductCenter、SystemConfig、MarketingCenter、DataAnalyticsHub 等 适合展示平台管理人员或运营人员在电脑上配置规则、看数据、做管理动作。
小程序端 偏移动端操作、代理侧/用户侧动作、快速查看和提交 MiniAppRender、MiniAgentPurchase、MiniListingConfigCenter、MiniMarketingCenter、MiniTaskCenter、MiniProductDetail 等 适合展示代理、运营人员或移动端角色在手机上选品、采购、推广、做任务。
说明：包内有 PC 端或小程序端组件，只能说明“有材料/有页面雏形”。是否可上线，还要继续看页面实跑、接口、保存回显和刷新保留。

三、选品中心和进货采购为什么不冲突

这两个词容易混在一起，但在现有包里不是同一个层级。

名称 真实含义 主责包 和另一个的关系
商品/选品中心 平台商品池、商品库、商品详情、商品审核或选择推广对象 平台配置为主；营销推广里只作为推广对象来源 它回答“有哪些商品可以看/选”。
进货采购 把选中的商品加入进货清单、生成采购单、形成已采购/库存/订单状态 平台配置_Copy 它回答“选中后怎么采购、下单、入库、上架”。
进货采购里的选品中心 是进货采购流程的第一个 Tab/入口，用来从商品池里挑货 平台配置_Copy 它不是另一个独立功能，而是进货采购流程的第一步。
所以不冲突：平台配置里的“商品/选品中心”偏商品池和平台管理；平台配置_Copy 里的“进货采购”包含一个“选品入口”，是采购流程第一步。

四、场景导航

这里不是说四个包一定按线性流程上线，而是用场景帮助展示各包边界。

1

先定平台规则

平台配置

先把平台规则、分润/模板、结算、供应商准入、系统设置等底层规则定好。

2

再把商品配置成可卖

平台配置_Copy

完成进货、订单、库存、上架配置、CPS 履约、批量上下架等动作。

3

然后做推广传播

营销推广

生成素材、配置活动、发推广任务、做裂变、视频号推客和推广数据回传。

4

最后每天运营跟进

日常运营

看今日任务、签到、标签维护、数据预警、活动执行进度和运营结果。

五、平台配置包

场景：平台要先把规则定好，后面的采购、上架、推广、运营才有统一口径。

平台规则中心
端侧 本包展示重点 典型证据/页面
PC 端 平台规则、商品库、分润模板、结算、系统设置、审核入口 MainApp、Dashboard、ProductCenter、SystemConfig、TemplateManager、EnhancedDashboard
小程序端 平台侧移动查看、商品详情、平台商品中心、审核详情等移动端入口 MiniAppRender、MiniPlatformProductCenter、MiniProductDetail、MiniProductAuditDetail

商品/选品中心（平台商品池） 主责

为什么存在
让平台知道商品从哪里来、怎么进入商品库、怎么查看详情和审核状态。
使用场景
平台侧看商品库、选品、商品详情、平台审核，以及进入后续配置入口。
不要混到哪里
不要展开成代理进货采购，也不要做完整库存入库。

分润配置 / 分润模板管理 主责

为什么存在
配置分润规则、模板、商品关联、生效范围和保存回显。
使用场景
代理商侧 D1/D2 分润、供应商/平台相关分润模板、默认模板、商品绑定模板等规则配置。
不要混到哪里
不要放进营销推广；不要在平台配置_Copy 里重做分润主页面。

佣金/结算 主责

为什么存在
让平台看清收益、结算、提现、结算状态和异常处理。
使用场景
平台查看收益概览、结算明细、提现入口和结算状态。
不要混到哪里
日常运营可以看数据入口，但不要替代结算系统。

供应链/云仓规则 规则

为什么存在
配置平台级供货、库存预警、云仓规则和默认仓储设置。
使用场景
平台定规则，例如库存预警、供货规则、默认仓库，而不是具体某个商品入库。
不要混到哪里
具体库存详情、手动入库/出库归平台配置_Copy。

批量操作规则 规则

为什么存在
定义平台商品批量配置规则，例如批量编辑配置、批量规则调整。
使用场景
平台统一调整商品配置规则、模板、标签或平台级字段。
不要混到哪里
代理侧批量上架、批量下架、批量补货归平台配置_Copy。

平台模式与系统设置 主责

为什么存在
平台需要统一入口来配置模式、系统设置、平台规则和角色边界。
使用场景
D1-D5 渠道代理模式、系统设置项、平台配置首页和入口管理。
不要混到哪里
不要把日常任务、采购库存、推广任务直接塞进系统设置。

供应商准入入口 入口/规则

为什么存在
平台配置里需要有供应商准入规则、邀请入口、审核入口。
使用场景
平台设置供应商能不能进入、怎么邀请、审核入口在哪里。
不要混到哪里
完整供应商入驻、订单、绩效、结算不在这四包范围内展开。

供应商商品管理入口 入口/审核

为什么存在
平台要能看到供应商提交的商品并做审核入口管理。
使用场景
平台商品审核、商品详情查看、审核规则入口。
不要混到哪里
供应商侧商品提交、上下架和履约不要放在平台配置包里展开。

六、平台配置_Copy 包

场景：商品已经被选中后，完成进货、上架、库存、订单和履约动作。

采购到上架工作台
端侧 本包展示重点 典型证据/页面
PC 端 采购/库存/上架配置相关管理页或数据看板入口 ProductCenter、AIOptimizationTracker、ProfitSharing、MainApp 类页面
小程序端 代理侧选品、进货清单、我的订单、上架配置、库存详情、手动入库/出库 MiniAgentPurchase、PurchaseListTab、MiniListingConfigCenter、StockDetailView、ManualStockInDialog、ManualStockOutDialog

进货采购（含选品入口） 主责

为什么存在
代理或平台要从商品池里选择商品，形成进货清单、采购单和已采购商品。
使用场景
进货采购里的选品入口、进货清单、已采购商品、进货订单、销售订单、下级订单。
边界
收藏、进货清单、已采购、库存要分清；不能用本地库存冒充真实采购闭环。

上架配置 主责

为什么存在
商品采购后，需要配置价格、渠道、库存、CPS、授权条件后才能正式开卖。
使用场景
待配置池、已配置待发布、已上架已开卖、组合套餐门槛礼包。
边界
这是商品开卖前配置，不是营销活动，也不是平台分润主页面。

我的订单 主责

为什么存在
采购和销售都会产生订单，需要能看状态、物流、确认收货、售后和入库。
使用场景
销售订单、进货订单、下级订单、待到货、待发货、已完成、售后。
边界
订单页不能只是演示列表，状态按钮必须有回显和状态流转。

CPS 履约 主责

为什么存在
有些商品不是批发进货，而是 CPS 推广，需要独立配置推广链路和履约状态。
使用场景
CPS 配置、CPS 订单演示、履约决策、状态回写。
边界
本包 CPS 偏订单履约和上架配置，不是营销推广里的任务中心。

收益/库存页 主责

为什么存在
商品上架和销售后，需要知道收益、库存数量、库存流水和库存详情。
使用场景
商品卡收益按钮、库存按钮、收益详情、库存详情、库存流水。
边界
不能用随机数或前端状态当真实库存，需要说明数据来源。

手动入库/出库 主责

为什么存在
库存需要人工调整时，必须有入库/出库原因、数量和流水记录。
使用场景
库存详情页里做手动入库、手动出库、备注、凭证、操作人和时间记录。
边界
不能只改前端 useState，需要形成库存流水。

批量操作 主责

为什么存在
已采购或库存商品很多时，需要批量上架、下架、补货、改价、设置 CPS。
使用场景
批量上架、批量下架、批量补货、批量 CPS 策略、批量改价。
边界
只在已采购/库存商品里生效；平台级批量规则归平台配置。

运营标签 共同能力

为什么存在
商品上架时需要带运营标签，后续运营、筛选、统计才能识别商品用途。
使用场景
单品上架配置里选择标签，批量标签管理里统一调整。
边界
平台配置_Copy 负责上架时设置标签；日常运营负责后续维护和回看。

七、营销推广包

场景：商品要被传播出去，代理要拿素材、做任务、发链接、看推广效果。

营销传播中心
端侧 本包展示重点 典型证据/页面
PC 端 营销中心、活动配置、素材管理、推广数据、官方营销管理 MarketingCenter、MarketingPromotion、PlatformMarketingPages、MaterialsPage、ProductCenter
小程序端 代理侧推广、素材领取、视频号推客、邀请海报、营销活动参与 MiniMarketingCenter、MarketingPromoAgentPage、MiniAICreativeCenter、MiniVideoFactory、MiniInvitePoster、MiniMarketingActivities

营销推广中心 主责

为什么存在
把素材、任务、活动、邀请海报、视频号推客和推广数据聚合在一个入口。
使用场景
代理或平台进入营销首页，选择要做推广、活动、素材还是看效果。
边界
不要放进货采购、库存入库、供应商订单。

AI 素材/智能工具 主责

为什么存在
推广需要文案、图片、视频、海报等素材，AI 工具负责生成和管理。
使用场景
生成素材、编辑素材、保存素材、投放到任务/活动、素材库复用。
边界
素材生成不是商品上架配置，也不是库存管理。

推广任务 主责

为什么存在
把推广动作变成任务，让代理知道今天该发什么、做什么、完成后有什么反馈。
使用场景
发海报、发视频、分享链接、完成转化、查看任务状态和推广结果。
边界
签到和日常活跃任务归日常运营；营销任务只保留推广动作。

裂变活动/邀请海报 主责

为什么存在
通过海报、邀请链路、分享链接做拉新和传播。
使用场景
生成海报、分享、访问/注册/购买、数据回传。
边界
要说明邀请对象是客户、代理还是供应商；供应商完整入驻不在这里展开。

视频号推客 主责

为什么存在
视频号推广需要素材、推广链接、发布记录、佣金效果。
使用场景
选素材、生成视频、绑定推广链接、发布记录、查看效果数据。
边界
这里的 CPS 是推广赚佣，不是平台配置_Copy 的订单履约。

营销活动 主责

为什么存在
平台或代理需要创建活动、配置规则、组织参与、查看执行效果。
使用场景
创建活动、配置规则、报名/参与、执行任务、查看效果。
边界
日常运营只跟进活动执行，不负责营销活动主创建链路。

商品/选品中心（推广对象来源） 来源

为什么存在
营销推广需要选择要推广的商品，所以要能看到商品和商品详情。
使用场景
选择推广商品、查看商品信息、生成对应素材或活动。
边界
只作为营销对象来源，不展开完整商品管理、进货采购和库存。

知识付费 推广对象

为什么存在
课程、学习入口、知识商品也可能需要营销推广。
使用场景
推广课程、学习入口、知识商品活动和转化链路。
边界
不要展开成完整课程系统后台，重点是推广和学习入口。
注意：营销推广包里出现进货采购、供应商订单、日常签到等痕迹时，先按“串包或外链入口”处理，不作为本包主开发功能。

八、日常运营包

场景：运营人员每天打开系统，知道今天要做什么、看什么、跟进什么。

运营工作台
端侧 本包展示重点 典型证据/页面
PC 端 运营工作台、数据分析、运营启动向导、活动执行跟进、模板选择 MainApp、PlatformOperationHub、OperationStartupWizard、DataAnalyticsHub、RealTimeDataBoard、TemplateMarket
小程序端 每日任务、签到、移动端运营入口、任务状态、运营展示 MiniHome、MiniTaskCenter、MiniAppRender、MiniAppShowcase、DailyCheckIn、DailyTasks

日常运营工作台 主责

为什么存在
运营人员需要一个首页，告诉他今天先做什么、哪些数据异常、哪些任务待完成。
使用场景
今日待办、数据预警、活动执行、标签维护、任务完成、运营启动向导。
边界
不要做成营销推广首页，也不要放采购库存主流程。

运营标签 共同能力

为什么存在
运营要根据标签看商品、人群、活动或代理状态，方便跟进和统计。
使用场景
运营视角维护标签、筛选标签、看标签统计、批量调整标签。
边界
平台配置_Copy 负责上架时设置标签；日常运营负责运营维护和回显。

日常任务 主责

为什么存在
把每日运营动作变成清单，让运营或代理知道做完没有、有没有奖励或反馈。
使用场景
任务名称、目标、完成按钮、完成状态、奖励/反馈、历史记录。
边界
推广动作任务归营销推广；这里负责日常活跃和运营动作。

签到 主责

为什么存在
签到用于记录活跃和连续运营动作，方便做激励或运营跟进。
使用场景
今日是否已签到、连续签到天数、奖励、历史记录。
边界
签到放日常运营工作台，不放营销推广中心。

数据分析中心 主责

为什么存在
运营需要看经营数据、商品数据、活动数据、任务数据，判断今天该做什么。
使用场景
经营数据、商品数据、活动数据、任务数据、时间筛选、详情下钻。
边界
没有真实数据源时不能写实时数据已接入。

活动执行 跟进

为什么存在
营销活动发起后，日常运营需要跟进执行进度、负责人、完成状态和数据反馈。
使用场景
活动执行清单、进度、负责人、完成状态、执行数据。
边界
营销推广负责创建和推广活动；日常运营负责执行跟进。

角色/等级管理 观察对象

为什么存在
运营需要知道哪些角色、代理、BOSS 等级需要跟进或培育。
使用场景
看成长状态、看谁需要跟进、看角色标签和等级表现。
边界
不要在这里配置分润系数或平台角色规则。

模板复制中心 运营模板

为什么存在
运营人员可以套用运营模板，快速启动类似活动、任务或运营方案。
使用场景
选择运营模板、复制模板、进入配置入口。
边界
完整系统搭建和模块组合不在这四包里展开。

十、展示判断规则

看到“规则、模板、分润、平台设置”

先归到 平台配置。因为这是平台规则，不是具体每天操作。

看到“进货、上架、库存、订单、CPS履约”

先归到 平台配置_Copy。因为这是商品从采购到可卖的执行链路。

看到“素材、推广、活动、裂变、视频号”

先归到 营销推广。因为这是对外传播和拉新转化。

看到“今日任务、签到、数据看板、运营跟进”

先归到 日常运营。因为这是每天运营人员打开后要做的事。
